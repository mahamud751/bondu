# SocialConnect implementation status

Last audited: 2026-07-24

This repository is a working MVP foundation, not a complete implementation of the full product specification. A feature is only marked implemented when both an API/data path and a usable mobile path exist.

**Complete product roadmap (better-than-BIGO):** see [COMPLETE_APP_ROADMAP.md](./COMPLETE_APP_ROADMAP.md).

| Area | Status | Current scope |

| Authentication | MVP | Phone OTP/password, verified email/password, and native Google and Apple entry points are implemented. Email registration and password recovery use hashed, expiring, attempt-limited single-use codes through Resend or a signed webhook; development codes never ship in production responses. The backend verifies social-provider signatures, issuer/audience/expiry/email claims, Apple nonce, and single-use server challenges; identities are uniquely linked. Refresh rotation, device sessions, suspicious-login alerts and activity are included. Every authenticated request rechecks current account status and role, immediately invalidating deleted accounts and stale privilege claims; suspended accounts retain only explicitly marked support and appeal access. |
| Profiles and social | MVP | Profiles, paginated/filterable discovery, language/interests/availability/rating/price filters, follow, connections and blocking are implemented. Service validation and PostgreSQL constraints reject unsafe amounts, negative category balances, uncovered reservations and inconsistent revenue splits. Configurable point conversion is snapshotted onto each payment so later setting changes cannot corrupt checkout, webhook, reconciliation or refund calculations. Live concurrent-spend integration/load tests remain. |

Stripe supports signed webhooks, atomic wallet crediting, compensated refunds at the original conversion snapshot, reconciliation, explicit dispute records, won/lost/chargeback transitions, Finance queues, risk restrictions and audit logs. ## Launch blockers

1. Validate native Agora voice/video and the signed Notification Service callback on physical devices/sandbox infrastructure, then finalize the production reconnect policy. The signed, idempotent callback and provider-triggered settlement path are implemented.
2. Configure and validate the production Twilio SMS, Persona identity-verification, Firebase push and private S3-compatible storage adapters. Their signed webhook and private-storage paths are implemented but require deployment credentials and live-provider acceptance.
3. Validate SSLCommerz against merchant sandbox/live acceptance and automate SSLCommerz refunds. Hosted checkout, validation, risk holds, IPN deduplication, settlement and reconciliation are implemented; Stripe dispute/chargeback ingestion and Finance visibility are implemented but require live webhook acceptance.
4. Validate the dedicated BullMQ worker deployment under load. The API owns repeatable schedulers while an independently scalable worker owns processing; provider settlement, call expiry, notifications, earning release and reconciliation use idempotency/retry records.
5. Validate ClamAV/object-storage behavior under production load and complete remaining security/audit coverage. Quarantine scanning, payout encryption, signed media URLs and global/auth-specific rate limits are implemented.
6. Complete operational monitoring validation, off-site backups, legal/privacy work and store compliance. Moderation appeals and fine-grained Admin RBAC are implemented but still require deployment-level policy review.

## Verified database invariants

The opt-in `npm run test:integration` suite uses `INTEGRATION_DATABASE_URL` and a real PostgreSQL database. Its 11 cases prove competing wallet reservations/debits, package and gift-card overspend prevention, atomic/idempotent package bonus crediting, exclusive active-call allowance use, reasoned/idempotent Admin adjustments, digital-gift and withdrawal idempotency, duplicate provider settlement, and unique webhook ingestion. On 2026-07-16 all 38 migrations, the deterministic seed (including a second idempotency run), and these concurrency tests passed against PostgreSQL 17.

## Definition of “MVP complete”

The product should not be described as production-ready until every launch blocker above is closed and the end-to-end flows are tested against real sandbox providers. The current local build is suitable for product validation and continued development.
